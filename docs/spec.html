Introduction

back to top

This is manual for SymOntoClay DSL.

This document contains all implemented grammar. You can use It.

SymOntoClay DSL is in development now. So new parts will be added here right after implementation and internal testing. I will try to do this as soon as possible.

Notation

back to top

The syntax is specified using Extended Backus-Naur Form (EBNF).

Source code representation

back to top

There are two views of source code file: as set of characters and as set of code entities.

Symbols

back to top

Characters

back to top

Source code is Unicode text encoded in UTF-8.

Newline = /* the Unicode code point U+000A */ .
UnicodeChar = /* an arbitrary Unicode code point except Newline */ .

Letters and digits

back to top

The underscore character _ (U+005F) is considered a letter.

Letter = "A" ... "Z" | "a" ... "z" | "_" .
DecimalDigit = "0" … "9" .

Kind of source code files

back to top

There are different kinds of source code files which can contain different code entities.

SourceCodeFileContent = WorldSourceCodeFileContent | NPCSourceCodeFileContent | HostSourceCodeFileContent | ClassSourceCodeFileContent .

World source code file

back to top

The file contains World declaration. There can be only one World declaration in the file. Also this file can contain other content which is allowed in Class source code file.

WorldSourceCodeFileContent = [ ClassSourceCodeFileContent ] [ WorldDecl ] [ ClassSourceCodeFileContent ] .

NPC source code file

back to top

The file contains NPC declaration. There can be only one NPC declaration in the file. Also this file can contain other content which is allowed in Class source code file.

NPCSourceCodeFileContent = [ ClassSourceCodeFileContent ] [ NPCDecl ] [ ClassSourceCodeFileContent ] .

Host source code file

back to top

The file contains Host declaration. There can be only one Host declaration in the file. Also this file can contain other content which is allowed in Class source code file.

HostSourceCodeFileContent = [ ClassSourceCodeFileContent ] [ HostDecl ] [ ClassSourceCodeFileContent ] .

Class source code file

back to top

The file contains all possible code entities, except World, NPC and Host declarations. There exist special source code files for these code entities. But content of Class source code file can be included into World, NPC and Host source code files.

ClassSourceCodeFileContent = [ ClassDeclSet ] [ FactDeclSet ] [ LogicRuleDeclSet ] .

Lexical elements

back to top

Comments

back to top

SymOntoClay supports both single and multiline comments. It is a programmer-readable explanation or annotation of source code like in many other programming languages. Also It helps to cut temporarily unused code fragments.

SingleLineComment = "//" [{ UnicodeChar }] NewLine .
//@r = @b = 1;
MultiLineComment = "/*" [{ UnicodeChar }] "*/" .
/*@r = @b = 1;
@bx >> @>log;*/

You can see examples here.

Tokens

back to top

Tokens form the vocabulary of the SymOntoClay DSL. There are four classes: identifiers, keywords, operators and punctuation, and literals.

Identifiers

back to top

Identifiers name code entities.

Since SymOntoClay has many kinds of code entities, I have decided It will be better to recognize kind of code entity by identifier. So SymOntoClay has many kinds of identifiers with different prefixes. Each of these identifiers has Its own meaning in code.

Identifier = ConceptIdentifier | InstanceIdentifier | VarIdentifier | SysVarIdentifier | QuestionVarIdentifier | LogicRuleVarIdentifier | ChannelIdentifier .

Concept identifier

back to top

Names a concept.

ConceptIdentifier = (( Letter { Letter | DecimalDigit } ) | ( "`" { UnicodeChar } "`")) .
a
_a
a2
aB
ab_1
Great_Britain
`Great Britain`

Instance identifier

back to top

Names an instance of concept.

InstanceIdentifier = "#" (( Letter { Letter | DecimalDigit } ) | ( "`" { UnicodeChar } "`")) .
#123

Variable identifier

back to top

Names a variable.

VarIdentifier = "@" (( Letter { Letter | DecimalDigit } ) | ( "`" { UnicodeChar } "`")) .
@a

System variable identifier

back to top

Names a system variable

SysVarIdentifier = "@@" (( Letter { Letter | DecimalDigit } ) | ( "`" { UnicodeChar } "`")) .
@@host

Question variable identifier

back to top

Names a variable for describing select-pattern in select query. This variable represents information which should be found in the select query.

QuestionVarIdentifier = "?" (( Letter { Letter | DecimalDigit } ) | ( "`" { UnicodeChar } "`")) .
?x

Logic rule variable identifier

back to top

Names special variable for Logic rule.

LogicRuleVarIdentifier = "$" (( Letter { Letter | DecimalDigit } ) | ( "`" { UnicodeChar } "`")) .
$x

Channel identifier

back to top

Names Channel.

ChannelIdentifier = "@>" (( Letter { Letter | DecimalDigit } ) | ( "`" { UnicodeChar } "`")) .
@>log

back to top

The following character sequences represent operators and punctuation:

&:}]=>
(:>{->
);{:>>
,?}

Integer literal

back to top

An integer literal is a sequence of digits representing an integer constant. Now integer literal has only decimal base.

IntegerLiteral = DecimalDigit [ { DecimalDigit } ] .
42

Floating-point literal

back to top

A floating-point literal is a decimal or hexadecimal representation of a floating-point constant.

A floating-point literal consists of an integer part (decimal digits), a decimal point and a fractional part (decimal digits). Now floating-point literal has only decimal base.

FloatPointLiteral = ( DecimalDigit [ { DecimalDigit } ] ) "." ( DecimalDigit [ { DecimalDigit } ] ) .
36.6

String literal

back to top

A string literal represents a string constant obtained from concatenating a sequence of characters.

StringLiteral = (""" UnicodeChar [{ UnicodeChar }] """) | ("'" UnicodeChar [{ UnicodeChar }] "'") .
"abc"
'abc'

You can see examples here.

Objects declaration

back to top

Everything is an object. The classical type-instance relationship is using, but there is not strict and fundamental distinction between the two.

Some objects have a special meaning. So Class represents a general concept which can have many instances.

Some objects have special engine support to make writing code easier and faster, or as entry point locators.

Class

back to top

Represents a general concept which can have many instances.

In general, It is similar to the concept 'class' in other OOP languages.

ClassDeclSet = ClassDecl [{ ClassDecl }] .
ClassDecl = "class" ConceptIdentifier [ InheritanceDecl ]
"{"
[ ClassBodyContent [{ ClassBodyContent }] ]
"}" .
ClassBodyContent = [ FactDeclSet ] [ LogicRuleDeclSet ] [ TriggersContent ] .
class exampleClass
{
}
class exampleClass is human, [0.1] animal
{
}

You can see examples here.

NPC

back to top

It is root object and entry point of NPC.

NPCIdentifier = ConceptIdentifier | InstanceIdentifier .
NPCDecl = "npc" NPCIdentifier [ InheritanceDecl ]
"{"
[ NPCBodyContent [{ NPCBodyContent }] ]
"}" .
NPCBodyContent = [ [ FactDeclSet ] [ LogicRuleDeclSet ] [ TriggersContent ] ] .
npc PeaceKeeper
{
    on Init =>
    {
    }
}
npc PeaceKeeper is human
{
    on Init =>
    {
    }
}

You can see examples here.

World

back to top

It is root object of World.

WorldIdentifier = ConceptIdentifier | InstanceIdentifier .
WorldDecl = "world" WorldIdentifier [ InheritanceDecl ]
"{"
[ WorldBodyContent [{ WorldBodyContent }] ]
"}" .
WorldBodyContent = [ FactDeclSet ] [ LogicRuleDeclSet ] .
world `Lost town`
{
}

Host

back to top

It is root object of Host.

HostIdentifier = ConceptIdentifier | InstanceIdentifier .
HostDecl = "host" HostIdentifier [ InheritanceDecl ]
"{"
[ HostBodyContent [{ HostBodyContent }] ]
"}" .

Waypoint

back to top

Represents a point in space, abstracting from the game engine's coordinate system.

Coordinates of points are presented in a more convenient way, similar to that is used by people in everyday life.

Now It is supported the coordinate system what is similar to Polar coordinate system. The radial coordinate is the distance from center of NPC to target. And the angular coordinate (in degrees only) starts from the sagittal plane of NPC in front of the NPC.

SymOntoClay engine provides mapping to the game engine's coordinate system.

WaypointDecl = "#@" "[" WaypointDistance "," WaypointAngleInDegrees "]" .
WaypointDistance = NumberExpr .
WaypointAngleInDegrees = NumberExpr .
#@[25, 30]

You can see examples here.

Fact

back to top

Represents knowledge in declarative way. It is an element of Logic programming.

Now the fact is a predicate sentence with small syntactic sugar for better inclusion into SymOntoClay DSL.

For example, expression "cat (#Alisa)" is equivalent to "#Alisa is a cat".

In the future, I am going to move away from the predicate form to a more human-readable one. In this case, the predicate form will be saved as an alternative way of describing.

FactDeclSet = FactDecl [{ FactDecl }] .
FactDecl = "{:" ( FactSection | FactSentence ) ":}" .
FactSection = [ ">:" ] "{" FactSentence "}" .
FactSentence = FactPredicate .
FactPredicate = ConceptIdentifier "(" FactPredicateParam [ { "," FactPredicateParam } ] ")" .
FactPredicateParam = ConceptIdentifier | InstanceIdentifier .
{: male(#Tom) :}
{: >:{ male(#Tom) } :}
{: parent(#Piter, #Tom) :}

You can see examples here.

Logic rule

back to top

Represents a logic conclusion rule. It allows to find and use facts which are not defined explicitly. It is an element of Logic programming.

A rule defines relation between two sets of facts during logic conclusion.

Logic searching uses depth-first search (DFS) with backtracking.

A rule consists of head (LogicRulePrimarySection) and body (LogicRuleSecondarySection). The head is true if the body is true. Both head and body are predicate sentences with small syntactic sugar for better inclusion into SymOntoClay DSL.

In the future, I am going to move away from the predicate form to a more human-readable one. In this case, the predicate form will be saved as an alternative way of describing.

Conjunctions can only appear in the body, not in the head of a rule. Only special variables can be used in the rule, the scope of which is limited by this rule.

Now logic searching happens when the select operator is only called explicitly. In the future, the area of using logic searching (conclusion) in SymOntoClay will be significantly expanded.

LogicRuleDeclSet = LogicRuleDecl [{ LogicRuleDecl }] .
LogicRuleDecl = "{:" LogicRulePrimarySection ImplicationSymbol LogicRuleSecondarySection ":}" .
LogicRulePrimarySection = [ ">:" ] "{" LogicRuleSentence "}" .
LogicRuleSecondarySection = "{" LogicRuleSentence "}" .
ImplicationSymbol = "->" .
LogicRuleSentence = LogicRulePredicate [ { AndOpExpr LogicRulePredicate } ] .
LogicRulePredicate = ConceptIdentifier "(" LogicRulePredicateParam [ { "," LogicRulePredicateParam } ] ")" .
LogicRulePredicateParam = ConceptIdentifier | InstanceIdentifier | LogicRuleVarIdentifier .
{: {son($x, $y)} -> { male($x) & parent($y, $x) } :}
{: >: {son($x, $y)} -> { male($x) & parent($y, $x) } :}

You can see examples here.

Common objects’ parts

back to top

The following parts can be in all or almost all kinds of objects.

Inheritance

back to top

SymOntoClay DSL has multiple fuzzy prototype-based inheritance.

That is, one object can have many “is a” relationships with other objects. A descendant object is related to a base object by the “is a” relationship, and is not its clone.

InheritanceDecl = "is" InheritanceDeclItem [ { "," InheritanceDeclItem } ] .
InheritanceDeclItem = [ "[" NumberExpr "]" ] ( ConceptIdentifier | InstanceIdentifier ) .

Additionally each relation has a float-point rank in range from 0 to 1. 1 is the highest rank, and 0 indicates the total absence of an "is a" relationship between these objects. By default the inheritance rank equals 1.

In operations using inheritance, the total inheritance rank between two objects is calculated by formula:

Rank total = ∏ i = 1 n Rank i

In this formula Rank i is inheritance rank of one link in the entire inheritance chain between two objects.

If inheritance has formed a rhombus, and there are several chains with different rank, then rank of chain with the minimum count of chain elements is taken.

Inheritance relationship can be read with operator "is", and It can be changed run time with "use is" statement.

Trigger

back to top

A trigger is a code that is automatically executed in response to certain events.

Now only Lifecycle triggers are available. But in the future I am going to develop other kinds of triggers.

TriggersContent = "on" LifecycleTriggerIdentifier "=>"
"{"
StatementsSet
"}" .

Life cycle triggers

back to top

Every object has a life cycle.

If there exist inheritance relation between object, then life cycle triggers of base object execute before the life cycle triggers of descendant object.

LifecycleTriggerIdentifier = InitLifecycleTriggerIdentifier .
Init

back to top

Initializes an object. It is like a constructor in other programming languages.

Init trigger of NPC is a first code which is executed for the NPC. So It is an entrypoint of the NPC.

InitLifecycleTriggerIdentifier = "Init" .

You can see examples here.

Expressions

back to top

An expression specifies the computation of a value by applying operators and methods to operands.

Expr = WaypointDecl | ConstExpr | VarExpr | SysVarExpr | ChannelExpr | AssignmentOpExpr | IsOpExpr | PointerOpExpr | LeftRightStreamOpExpr | SelectOpExpr | InsertOpExpr .

Constant expression

back to top

Constant expressions may contain only literals and are evaluated at compile time.

ConstExpr = ConceptIdentifier | InstanceIdentifier | VarIdentifier | SysVarIdentifier | ChannelIdentifier | StringLiteral | NumberExpr .
NumberExpr = IntegerLiteral | FloatPointLiteral .
25
30.2
"abc"

Variable

back to top

A variable is a storage location for holding a value.

Both reading and writing are allowed for variables.

Default value of variable is NULL.

VarExpr = VarIdentifier .
@x
@`target value`

You can see examples here.

System variable

back to top

It's a special variable, which value is only written by engine.

Only reading is allowed for system variables.

SysVarExpr = SysVarIdentifier .
@@host

@@host

back to top

Contains a link to the C# methods of this NPC which are defined at the Unity3D level of the NPC logic.

Allows to call low-level C# methods for interaction with Unity3D.

It does nothing when It launched with the "CLI run".

Assignment operators

back to top

Operator "="

back to top

Associativity: Right-to-left.

Writes a value from the right operand to the left operand and returns the value of the right operand, which can be used in further calculations.

The left operand must be a variable.

AssignmentOpExpr = Expr "=" Expr .
@x = 1;
@r = @b = 1;

Relational operators

back to top

Operator "is"

back to top

Returns inheritance rank. The left operand contains a checked object. The right-hand operand contains a possible base object.

IsOpExpr = ( ConceptIdentifier | InstanceIdentifier ) "is" ( ( [ "[" NumberExpr "]" ] ( ConceptIdentifier | InstanceIdentifier ) ) | ( [ "not" ] ( ConceptIdentifier | InstanceIdentifier ) ) ) .
exampleClass is human >> @>log;

Using "not" returns 1 if inheritance relationship is not exist between two operands otherwise returns 0.

exampleClass is not human >> @>log;

You can see examples here.

Logic operators

back to top

Operator "&"

back to top

Оperator of logical conjunction.

AndOpExpr = Expr "&" Expr.
male($x) & parent($y, $x)

Member access operators

back to top

Operator "."

back to top

Gets a reference to a member (specified in the right operand) of an object (specified in the left operand).

The right-hand operand must contain the member's identifier.

PointerOpExpr = Expr "." Expr .
@@host.`go`

Call operators

back to top

Operator "()"

back to top

Calls an object (or member reference) with parameters specified in round brackets.

Whether a call is synchronous or asynchronous is determined by the presence of a "~" before the opening round bracket of the parameter list.

CallOpExpr = Expr [ "~" ] "(" [ NamedParamsOfCallOpExprSet ] ")" .
NamedParamsOfCallOpExprSet = NamedParamOfCallOpExpr [ { "," NamedParamOfCallOpExpr } ] .
NamedParamOfCallOpExpr = ConceptIdentifier ":" Expr .

A synchronous call is made when there is no "~". In this case, the operator returns directly the result of the called method.

@@host.`go`(to: #@[25, 30]);

You can see examples here.

An asynchronous call is made when "~" is present. In this case, the operator returns a reference to the object representing the asynchronous operation. Working with asynchronous operations will be designed in details in the future.

@@host.`go`~(to: #@[25, 30]);

You can see examples here.

Stream operators

back to top

Operator ">>"

back to top

Copies data from a source (SourceOfStreamOpExpr) to a destination DestOfStreamOpExpr.

The source (SourceOfStreamOpExpr) cannot be a channel. The destination (DestOfStreamOpExpr) must only be a channel.

In the future I am going to extend the functionality of this operator.

SourceOfStreamOpExpr = Expr .
DestOfStreamOpExpr = Expr .
LeftRightStreamOpExpr = SourceOfStreamOpExpr ">>" DestOfStreamOpExpr .
"End" >> @>log;
select {: son(?x, ?y) :} >> @>log;

Data source operators

back to top

select (Operator "?")

back to top

Performs a logical search and returns an object which contains the results of the search.

Now the result of a logical search can only be logged for demonstrating this search. In the future, I am going to develop means for practical using results of logical searching in game development.

SelectOpExpr = ( "select" | "?" ) ( FactDecl | SelectQuery ) .
SelectQuery = "{:" ( SelectQuerySection | SelectQuerySentence ) ":}" .
SelectQuerySection = ">:" "{" SelectQuerySentence "}" .
SelectQuerySentence = SelectQueryPredicate .
SelectQueryPredicate = ( ConceptIdentifier | QuestionVarIdentifier ) "(" SelectQueryPredicateParam [ { "," SelectQueryPredicateParam } ] ")" .
SelectQueryPredicateParam = ConceptIdentifier | InstanceIdentifier | QuestionVarIdentifier .

Searched values are defined by the question variables in the searching query.

select {: son(?x, ?y) :} >> @>log;
<yes>
?y = #piter; ?x = #tom

You can see examples here.

? {: son(?x, ?y) :} >> @>log;
<yes>
?y = #piter; ?x = #tom

You can see examples here.

Question variables can be used for searching predicates.

? {: ?z(#Alisa_12, ?x) :} >> @>log;
<yes>
?z = can(bird,fly); ?x = fly

You can see examples here.

In the absence of question variables, only general result will be obtained: is it possible or not to find such facts in the knowledge database.

? {: can(#Alisa_12, fly) :} >> @>log;
<yes>

You can see examples here.

insert

back to top

Writes a fact into the knowledge storage of current NPC. Returns the added fact.

InsertOpExpr = "insert" FactDecl .
insert {: >: { bird (#1234) } :};

You can see examples here.

Statements

back to top

Statements control execution.

StatementsSet = Statement [{ Statement }] .
Statement = ExpressionStatement | UseIsOpExpr .

Expression statement

back to top

Executes an expression line.

ExpressionStatement = Expr ";" .
@@host.`go`~(to: #@[25, 30]);

"use is" statement

back to top

Sets an inheritance relationship ("is a") between two objects.

UseIsOpExpr = "use" ( ConceptIdentifier | InstanceIdentifier ) "is" ( ( [ "[" NumberExpr "]" ] ( ConceptIdentifier | InstanceIdentifier ) ) | ( [ "not" ] ( ConceptIdentifier | InstanceIdentifier ) ) ) .
use @@self is linux;
use exampleClass is human;
use exampleClass is [0.5] human;
use #`Alisa 12` is [0.6] human;

Using "not" removes the inheritance relationship.

use exampleClass is not human;

You can see examples here.
